# Принцип единственной ответственности (Single Responsibility Principle — SRP)

## Определение
**Принцип единственной ответственности (SRP)** — это первый из пяти принципов SOLID. Он гласит: 
> *Класс должен иметь только одну причину для изменения.*

Это означает, что каждый класс должен отвечать за выполнение только одной логической функции в системе.

## 1. Пример нарушения принципа
В этом примере класс `Order` перегружен обязанностями: он и считает стоимость заказа, и отвечает за сохранение данных в базу данных.

```python
class Order:
    def __init__(self, items):
        self.items = items

    def calculate_total(self):
        """Логика расчета стоимости заказа."""
        return sum(item.price for item in self.items)

    def save_to_database(self):
        """Логика работы с БД (НАРУШЕНИЕ SRP)."""
        print(f"Сохраняем заказ в базу данных...")
Почему здесь нарушен SRP?
Класс имеет две причины для изменений:

Если изменится бизнес-логика расчета цен (например, добавление скидок).

Если изменится способ хранения данных (например, переход с SQL на облачное API).

2. Исправление (Рефакторинг)
Чтобы исправить нарушение, мы разделяем ответственности. Класс Order остается только для бизнес-логики, а класс OrderRepository берет на себя всю работу с хранилищем.

Python
# Класс отвечает только за данные и бизнес-логику заказа
class Order:
    def __init__(self, items):
        self.items = items

    def calculate_total(self):
        return sum(item.price for item in self.items)

# Класс отвечает только за сохранение (инфраструктурный уровень)
class OrderRepository:
    def save(self, order: Order):
        print(f"Сохраняем заказ в базу данных...")
3. Комментарий к исправлению
Разделение ответственности: Теперь класс Order ничего не знает о способах хранения. Если мы решим сохранять заказы не в базу, а в CSV-файл, нам не придется менять код самого заказа.

Удобство тестирования: Мы можем легко протестировать расчет стоимости Order без необходимости имитировать (мокать) подключение к базе данных.

Поддерживаемость: Код стал более модульным. Мы соблюдаем принцип SOLID, обеспечивая чистоту архитектуры, где изменения в инфраструктуре не влияют на бизнес-логику приложения.
